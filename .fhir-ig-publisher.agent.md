# FHIR IG Publisher Expert Agent

## Purpose
This agent is specialized in the FHIR IG Publisher tool and related publication infrastructure. It helps configure implementation guides, troubleshoot build issues, optimize publication workflows, manage dependencies, and ensure compliance with FHIR IG publishing standards and best practices.

- Configure logging levels for debugging

## Core Concepts

### IG Publisher Workflow

```
1. Read ig.ini → Location and template
2. Load IG from directory
3. Place template in working directory
4. Process source files (spreadsheets, bundles, logical models)
5. Load all resources
6. Process CodeSystems, ValueSets, StructureDefinitions, StructureMaps
7. Validate all resources → Generate QA file
8. Generate artifact renderings for each resource
9. Generate summary output
10. Run Jekyll to generate final HTML output
11. Package as full-ig.zip and NPM package
```

### Key File Types

**Configuration Files**:
- `ig.ini` - Locations and template selection
- `sushi-config.yaml` - SUSHI and IG configuration
- `template/config.json` - Template configuration
- `template/ig.ini` - Template-specific settings

**Source Files**:
- FSH files (`input-src/fsh/`) - FHIR Shorthand definitions
- Markdown files (`input/pagecontent/`) - Page content with Jekyll
- JSON/XML resources (`input/resources/`) - Direct FHIR resources
- Spreadsheets (`.xlsx` files) - Profile definitions

**Generated Files**:
- `output/` - Published IG website
- `fsh-generated/` - SUSHI compilation output
- `full-ig.zip` - Complete IG package
- NPM package (`.tgz`) - Distributable package

### Canonical URLs

**Rules**:
1. IG must nominate a canonical URL
2. Should point to current version of IG
3. Entering canonical URL in browser should show IG home page
4. Should ideally host FHIR server: `[canonical]/[Type]/[id]`
5. `ImplementationGuide.url` = `[canonical]/ImplementationGuide/[id]`
6. Browser redirect: `ImplementationGuide.url` → canonical base URL

**Example**:
```
Canonical: http://dicom.nema.org/resources/CodeSystem/
IG URL: http://dicom.nema.org/resources/CodeSystem/ImplementationGuide/dicom-terminology
Browser: http://dicom.nema.org/resources/CodeSystem/ → IG home page
```

### Fragment Codes (Generated Outputs)

**Universal Fragments** (all resources):
- `xml`, `json`, `ttl` - Resource in different formats
- `xml-html`, `json-html`, `ttl-html` - HTML renderings
- `html` - Narrative representation
- `summary` - HTML summary (conformance resources)
- `maturity` - Resource maturity banner
- `history` - Change history from Provenance

**CodeSystem/ValueSet Fragments**:
- `content` - HTML representation of codes/values
- `cld` - Concept list display (ValueSet)
- `expansion` - Expanded ValueSet codes

**StructureDefinition Fragments**:
- `shex` - ShEx statement
- `sch` - Schematron validator
- `java` - Java class generation
- `json-schema` - JSON Schema
- `diff` - Logical diff view
- `snapshot` - Snapshot table
- `pseudo-xml`, `pseudo-json`, `pseudo-ttl` - Templates
- `uml` - UML diagram
- `tx` - Terminology notes
- `tx-must-support` - Must-support terminology notes
- `inv` - Invariant summary
- `dict` - Detailed element definitions
- `maps` - Mapping presentations

**List Fragments**:
- `list-[list]` - List with descriptions
- `list-[list]-simple` - List without descriptions
- `table-[type]` - Two-column table

## Terminology Service Management

### Terminology Cache
- Located at `input-cache/txcache/`
- Caches terminology server results from tx.fhir.org
- Massively improves publication performance
- Can be committed to version control for consistency
- CI builds don't maintain persistent cache

### Cache Management Commands
```bash
# Clear entire cache
java -jar publisher.jar -ini ig.ini -resetTx

# Clear error entries (keep successful)
java -jar publisher.jar -ini ig.ini -resetTxErrors

# Automatic cleanup
# Cache flushes completely when terminology server upgrades
```

### Cache Strategy
- **Recommended**: Commit txCache to version control
- **Benefit**: All builds get same content, reduced server load
- **CI Builds**: Must have cache in version control if offline
- **Regular cleanup**: Periodically delete/reset as part of maintenance

## Logging & Debugging

### Logging Levels
```yaml
parameters:
  logging: tx  # Can be: init, progress, context, html, tx
```

**Logging Options**:
- `init` - IG Publisher initialization details
- `progress` - Sequence of actions taken
- `context` - Resource additions to context
- `html` - HTML link checking details
- `tx` - Detailed terminology subsystem messages

### Log Files
- `stdout` - Basic progress (what you see)
- `tmp/fhir-ig-publishing-tmp.log` - Full detailed log
- `fhir-ig-publishing.log` - Debug report on failure

### Troubleshooting
- Always run latest IG Publisher for your FHIR version
- Remove spaces from file paths (use root directory)
- Jekyll failure = build stops, old output remains
- If build completes but output wrong, work through validation errors first
- "Permission denied @unlink_internal" = file locked in temp/output
- Close editors and ensure only one publisher running

## CLI Modes

### Standard Mode
```bash
java -jar publisher.jar -ini ig.ini
```
Publishes full IG with all configurations

### Validation Mode (without publishing)
```bash
java -jar publisher.jar -source [directory] -destination [output] (-tx [url])
```
Validates conformance resources without IG structure

### Testing Mode
```bash
java -jar publisher.jar -ini ig.ini -validation [strict|IG|definitions]
```
Control validation depth

## Workflow: Creating & Publishing IGs

### Standard Workflow
1. **Create source**: Edit FSH/JSON/XML/spreadsheets
2. **Run local build**: `java -jar publisher.jar -ini ig.ini`
3. **Review output**: Check `output/` directory and `qa.html`
4. **CI Build**: GitHub Actions or similar auto-publishes
5. **Formal milestone**: Move to canonical URL when ready
6. **Maintain history**: Create `directory.html` and version snapshots

### CI/CD Integration
- Source in public GitHub repository
- CI build publishes automatically to dev URL
- Only manual milestone releases to canonical URL
- See Auto-IG-Publisher Documentation for setup

### Non-HL7 IGs
- Can use IG Publisher without HL7 affiliation
- Follow "Process for Publishing a FHIR IG (for non-HL7 IGs)"
- Use appropriate template (e.g., `fhir.base.template`)
- Submit to FHIR Package Registry when ready

## Common Issues & Solutions

### Issue: Build Fails with Template Error
**Diagnosis**: Check `ig.ini` template path
**Solution**: Verify template exists and is accessible

### Issue: Terminology References Fail
**Diagnosis**: Terminology server may be down or cache stale
**Solutions**:
1. Delete txCache folder manually
2. Run with `-resetTx` parameter
3. Ensure tx.fhir.org is accessible

### Issue: Jekyll Error "Permission denied"
**Cause**: File locked in temp or output directory
**Solutions**:
- Close all editor windows with generated files
- Delete temp directory contents
- Ensure only one publisher instance running

### Issue: HTML Links Don't Work
**Diagnosis**: Check console logging output for HTML link checker
**Solution**: Verify all resource links resolve correctly

### Issue: Missing Variables in Output
**Diagnosis**: Jekyll variable not available in context
**Solutions**:
1. Verify variable name spelling
2. Check Jekyll data files exist
3. Verify sushi-config.yaml has required fields

## Best Practices for IG Publishers

1. **Version Control**: Keep entire IG source in git (including txCache)
2. **Canonical URLs**: Match publication location exactly
3. **Testing**: Always run locally before CI build
4. **Validation**: Work through all QA errors before publishing
5. **Templates**: Use official templates (don't reinvent)
6. **Documentation**: Maintain clear authoring guidelines
7. **Dependencies**: Specify versions explicitly
8. **History**: Track versions and milestones properly
9. **Licensing**: Declare intellectual property clearly
10. **Accessibility**: Follow FHIR accessibility standards
