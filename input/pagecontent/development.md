## Architecture Overview

This Implementation Guide is built through an integrated pipeline that combines:

1. **DICOM Specification** → Source of all terminology
2. **Python Tools** → Automated extraction and conversion from DICOM sources
3. **FHIR Shorthand (FSH)** → Human-readable resource definitions
4. **SUSHI** → Compilation of FSH into FHIR JSON resources
5. **FHIR IG Publisher** → Generation of published IG website
6. **Jekyll** → Final HTML rendering for web presentation

## Directory Structure & File Purposes

|---|---|

| `sushi-config.yaml` | **Critical Config** - IG metadata, dependencies, parameters |

```
├── ig.ini                    # IG Publisher entry point

    └── pagecontent/          # the pages in this specification
```

### Python Tools (`dicom_tools/`)

**Purpose**: Automated extraction and generation of DICOM terminology as FHIR resources

| Module | Purpose |
|---|---|
| `createDicomFhirResourcesFromDocBook.py` | **Main generator** - Creates CodeSystems and ValueSets from DICOM DocBook |
| `downloadDicomValueSets.py` | Downloads official DICOM ValueSets |
| `uid_main.py` | Processes DICOM UIDs (identifiers) |
| `sop_classes.py` | Processes DICOM SOP Classes |
| `cid_valuesets.py` | Processes DICOM Context ID ValueSets |
| `data_elements.py` | Processes DICOM Data Elements (tags) |
| `vr_table.py` | Processes DICOM Value Representations |
| `anatomical_region_and_body_part.py` | Anatomical region terminology |
| `parseDocbook.py` | Parser for DICOM DocBook XML format |
| `doc_book_tools.py` | Utilities for DocBook processing |

**Input**: DICOM specification files (downloaded from `dicom.nema.org`)  
**Output**: FSH files or direct JSON FHIR resources

### FSH Source (`input-src/fsh/`)

**Purpose**: Human-readable FHIR Shorthand definitions of all CodeSystems and ValueSets

| File Pattern | Content |
|---|---|
| `CodeSystem-dicom-dcm-*.fsh` | DCM codes (main DICOM codes - very large, 21000+) |
| `CodeSystem-dicom-*.fsh` | Other DICOM CodeSystems (UIDs, SOP classes, etc.) |
| `ValueSet-dicom-*.fsh` | Curated value sets |

**Key Convention**: FSH files are the primary source for resource definitions and are manually curated or auto-generated.

### Generated Resources (`input/resources/`)

**Purpose**: FHIR JSON resources downloaded or generated from external sources

| Type | Content |
|---|---|
| `CodeSystem-dicom-*.json` | Downloaded DICOM CodeSystems |
| `ValueSet-*.json` | Downloaded DICOM ValueSets |
| Generated resources | Output from Python tools |

### Page Content (`input/pagecontent/`)

**Purpose**: Documentation pages for the IG website

| File | Content |
|---|---|

**Format**: Markdown with YAML front matter - Jekyll processes these into HTML

## Build Process & Commands

### Standard Build (Recommended)

```bash
./_genonce.sh          # On Unix/Mac
_genonce.bat           # On Windows
```

**What happens**:
1. Reads `ig.ini` → Locates IG and template
2. Runs SUSHI compiler on `input-src/fsh/` → Produces `fsh-generated/`
3. Loads all resources (from `input-src/fsh/`, `fsh-generated/`, `input/resources/`)
4. Validates all resources → Produces `output/qa.html`
5. Generates resource pages (HTML, JSON, XML formats)
6. Runs Jekyll to render pages
7. Packages as `full-ig.zip` and NPM package

**Output**: `output/` directory contains the final IG website

### Python Generation Workflow

```bash
./runPythonAndBuildIg.sh
```

**What happens**:
1. Runs Python tools in `dicom_tools/` (read DICOM sources)
2. Generates intermediate FSH or JSON files
3. Proceeds with standard build

**When to use**: After updating DICOM specification sources or changing terminology generation rules

### SUSHI-Only Compilation

```bash
./runSushiAndBuildIg.sh
```

**What happens**:
1. Runs SUSHI on `input-src/fsh/` → `fsh-generated/`
2. Runs IG Publisher to publish

**When to use**: After editing FSH files; faster than full Python generation

### Continuous Build (Development)

```bash
./_gencontinuous.sh
```

**What happens**:
- Watches for file changes
- Auto-rebuilds when you edit FSH files, pages, or configuration
- Great for live development

**Output**: Rebuilds `output/` automatically

### Debugging Build Issues

```bash
# Check IG Publisher config
cat ig.ini

# View SUSHI configuration
cat sushi-config.yaml

# Check validation errors
open output/qa.html          # Mac
xdg-open output/qa.html      # Linux
start output/qa.html         # Windows

# Check build logs
tail temp/fhir-ig-publishing-tmp.log
```

## Content Sources

### Primary Source: DICOM Specification

**Official DICOM Standard**: [https://dicom.nema.org/](https://dicom.nema.org/)

**What we use**:
- **DICOM DocBook format** (XML) - Official specification document
- **DICOM Unique Identifiers (UIDs)** - Hierarchical naming system
- **DICOM Data Elements** - Tags with numeric identifiers and keywords
- **DICOM Value Representations (VR)** - Data type definitions
- **SOP Classes** - Service Object Pair classes for DICOM services
- **Context ID (CID) ValueSets** - Predefined coded concepts
- **Anatomical Regions & Body Parts** - Domain-specific terminologies

### How Python Tools Access DICOM Content

```
DICOM Specification Files (dicom.nema.org)
           ↓
Downloaded by: downloadDocbook.sh
           ↓
Stored in: dicom.nema.org/ directory
           ↓
Parsed by: dicom_tools/*.py
           ↓
Converted to: CodeSystems & ValueSets (FSH or JSON)
           ↓
Compiled by: SUSHI
           ↓
Published by: FHIR IG Publisher
```

### DICOM Sources Used

| Content | Source | Python Module |
|---|---|---|
| **Data Elements** | DICOM docbook (PS3.6) | `data_elements.py` |
| **UIDs** | DICOM docbook (PS3.10) | `uid_main.py` |
| **SOP Classes** | DICOM docbook | `sop_classes.py` |
| **Value Reps** | DICOM docbook (PS3.5) | `vr_table.py` |
| **CID ValueSets** | DICOM Standard | `cid_valuesets.py` |
| **Anatomical Terms** | DICOM/SNOMED | `anatomical_region_and_body_part.py` |

### Downloading Fresh DICOM Content

```bash
./downloadDocbook.sh
```

**What it does**:
1. Downloads latest DICOM DocBook XML from `dicom.nema.org`
2. Stores in `dicom.nema.org/` directory
3. Makes available for Python tools to parse

**When to run**: When you want fresh DICOM specification content

### Version Control Strategy

**Committed to Git** (always have these):
- `input-src/fsh/` - FSH source definitions
- `sushi-config.yaml`, `ig.ini` - Configuration
- `input-cache/txcache/` - Terminology service cache (for reproducible builds!)
- `.fshignore` - FSH compiler exclusions

**NOT committed** (generated, can be regenerated):
- `fsh-generated/` - Output from SUSHI
- `output/` - Output from IG Publisher
- `temp/` - Build artifacts
- `dicom.nema.org/` - Downloaded DICOM sources

### Terminology Service Caching

The FHIR IG Publisher uses **tx.fhir.org** (terminology server) for validating codes and ValueSet expansions.

```
Terminology Service: tx.fhir.org
           ↓
Results cached in: input-cache/txcache/
           ↓
Strategy: Commit txcache/ to git for reproducible builds
```

**Benefits of committing cache**:
- All builds get identical terminology results
- CI builds work offline
- Reduced server load
- Reproducible builds

## Development Workflow

### For Resource Definition (FSH)

```
1. Edit: input-src/fsh/*.fsh
2. Run: ./runSushiAndBuildIg.sh
3. Check: output/qa.html for errors
4. Review: output/CodeSystem-*.html (generated version)
5. Iterate: Fix FSH, rebuild
```

### For Page Content

```
1. Edit: input/pagecontent/*.md
2. Run: sushi .  (just compile FSH, no FHIR publisher)
3. Or: ./_genonce.sh (full build)
4. Check: output/*.html
5. Iterate: Fix pages, rebuild
```

### For Python Terminology Generation

```
1. Update: Python code in dicom_tools/
2. Run: ./downloadDocbook.sh (get fresh DICOM)
3. Run: ./runPythonAndBuildIg.sh (full pipeline)
4. Check: output/qa.html
5. Review: Generated CodeSystems/ValueSets
6. Iterate: Fix generation logic, rebuild
```

### Before Publishing

```
1. Run full build: ./_genonce.sh
2. Check output/qa.html for errors
3. Verify all resources rendered
4. Test pages in browser
5. Review full-ig.zip contents
6. Commit to git
7. Create release/tag
8. Upload to canonical URL
```
